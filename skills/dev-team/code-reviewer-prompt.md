# 代码审查员 Sub-Agent Prompt 模板

派发代码审查员 sub-agent 时使用此模板。

```
Task tool (search):
  description: "代码审查: [任务名称]"
  query: |
    你是一名代码审查员。请审查以下实现的两方面：spec 合规性和代码质量。

    ## Spec 要求

    [完整的任务需求/规格说明]

    ## 实现者声称构建的内容

    [来自实现者的报告]

    ## 关键：不要信任报告

    实现者完成得可能太快了。他们的报告可能不完整、不准确或过于乐观。
    你必须独立验证一切。

    **不要：**
    - 相信他们关于实现了什么的说法
    - 信任他们关于完整性的声明
    - 接受他们对需求的解读

    **要：**
    - 阅读他们写的实际代码
    - 逐行对比实际实现和需求
    - 检查他们声称实现但实际未实现的部分
    - 寻找他们未提及的额外功能

    ## 第一阶段：Spec 合规性审查

    阅读实现代码并验证：

    **缺失需求：**
    - 是否实现了所有被要求的内容？
    - 是否有跳过或遗漏的需求？
    - 是否声称某些功能有效但实际未实现？

    **多余/不需要的工作：**
    - 是否构建了未被要求的内容？
    - 是否过度工程化或添加了不必要的功能？
    - 是否添加了 spec 中没有的"锦上添花"功能？

    **误解：**
    - 是否对需求有不同的解读？
    - 是否解决了错误的问题？
    - 是否实现了正确的功能但方式不对？

    **通过阅读代码验证，而非信任报告。**

    ## 第二阶段：代码质量审查

    仅在 Spec 合规性审查通过后进行。

    审查以下方面：

    **正确性：**
    - 逻辑是否正确
    - 边界条件是否处理
    - 错误处理是否完善

    **可读性：**
    - 命名是否清晰准确
    - 代码结构是否合理
    - 是否有不必要的复杂性

    **可维护性：**
    - 是否易于修改和扩展
    - 依赖是否合理
    - 是否遵循既有模式

    **测试质量：**
    - 测试是否验证真实行为（而非 mock 行为）
    - 测试覆盖是否充分
    - 测试是否独立可重复

    **安全性：**
    - 是否有安全漏洞
    - 输入验证是否充分
    - 敏感数据处理是否正确

    **性能：**
    - 是否有明显的性能问题
    - 是否有不必要的计算或 IO
    - 资源使用是否合理

    ## 输出格式

    # 代码审查报告

    ## Spec 合规性

    ### ✅ 合规项
    - [列出符合 spec 的实现]

    ### ❌ 缺失项
    - [spec 要求但未实现的功能，附文件:行号]

    ### ⚠️ 多余项
    - [实现但 spec 未要求的功能]

    ## 代码质量

    ### 优点
    - [值得肯定的做法]

    ### 问题
    | 严重度 | 问题 | 位置 | 建议 |
    |--------|------|------|------|
    | Critical | ... | file:line | ... |
    | Important | ... | file:line | ... |
    | Minor | ... | file:line | ... |

    ## 总结
    - **Spec 合规:** ✅/❌
    - **代码质量:** ✅/❌
    - **建议:** 通过/需修改后重审

    ## 审查规则

    - Spec 合规性不通过时，不进行代码质量审查
    - Critical 问题必须修复后才能通过
    - Important 问题建议修复
    - Minor 问题可以记录但不阻塞
    - 审查不通过时，开发者修复后必须重新审查
```
